# کیپسٹون پروجیکٹ ٹربول شوٹنگ گائیڈ

یہ گائیڈ کیپسٹون پروجیکٹ کے لیے متعدد پیچیدہ سسٹم (ROS 2، شبیہ سازی، AI ماڈلز) کو یکجا کرنے کے دوران سامنے آنے والے عام مسائل کو حل کرتی ہے۔ اس کو تشخیص اور مسائل کو حل کرنے کے لیے ایک حوالہ کے طور پر استعمال کریں۔

## جنرل ROS 2 مواصلت کے مسائل

### علامات
*   نوڈز مواصلت نہیں کر سکتے (ٹاپکس/سروسز منسلک نہیں ہیں)۔
*   "Failed to contact master" یا "No route to host" جیسے خرابی کے پیغامات (اگر متعدد مشین سیٹ اپ استعمال کر رہے ہیں)۔
*   `ros2 topic list` یا `ros2 node list` متوقع سے کم اشیاء دکھاتا ہے۔

### حل
1.  **`ROS_DOMAIN_ID` چیک کریں**: یقینی بنائیں کہ ROS 2 ماحول کو سورس کرنے والے تمام ٹرمنل (اور ممکنہ طور پر Isaac Sim اگر یہ ROS 2 کے ساتھ براڈج کرتا ہے) ایک ہی `ROS_DOMAIN_ID` استعمال کر رہے ہیں۔ ڈیفالٹ 0 ہے۔
    ```bash
echo $ROS_DOMAIN_ID # موجودہ ID چیک کریں
export ROS_DOMAIN_ID=0 # ٹرمنل سیشن میں 0 (یا مطلوبہ ID) پر سیٹ کریں
    ```
2.  **نیٹ ورک کنفیگریشن کی تصدیق کریں**: اگر آپ کے سسٹم کے حصے مختلف مشینوں پر یا مختلف ڈوکر کنٹینرز میں چل رہے ہیں، تو یقینی بنائیں کہ نیٹ ورک کی ترتیبات ادراک کی اجازت دیتی ہیں (مثلاً ڈوکر میں `--network=host` استعمال کرنا، یا `RMW_IMPLEMENTATION` اور `ROS_LOCALHOST_ONLY` کنفیگر کرنا)۔
3.  **فائر وال چیک کریں**: یقینی بنائیں کہ فائر وال کے قواعد ROS 2 کے ذریعہ استعمال کردہ DDS (ڈیٹا ڈسٹری بیوشن سروس) ٹریفک کو بلاک نہیں کر رہے ہیں۔ پورٹس عام طور پر 7400 سے اوپر کی رینج میں ہوتی ہیں۔
4.  **لاگز کا معائنہ کریں**: ROS 2 نوڈز کے کنسول آؤٹ پٹ کو مخصوص خرابی کے پیغامات کے لیے چیک کریں۔ زیادہ ورbose لاگنگ کے لیے `ros2 run <pkg_name> <node_name> --ros-args --log-level debug` استعمال کریں۔

## شبیہ سازی (Gazebo/Isaac Sim/Unity) - ROS 2 براڈج کے مسائل

### علامات
*   شبیہ ROS 2 کمانڈز کا جواب نہیں دیتا۔
*   سینسر ڈیٹا (مثلاً کیمرہ امیجز، LiDAR) شبیہ سازی سے ROS 2 ٹاپکس پر شائع نہیں ہوتا۔
*   روبوٹ ماڈل شبیہ میں صحیح طریقے سے نہیں دکھائی دیتا یا کنٹرول بے ترتیب ہے۔

### حل
1.  **پلگ ان انسٹالیشن کی تصدیق کریں**: یقینی بنائیں کہ ضروری ROS 2 براڈج پلگ انز انسٹال اور لوڈ ہیں۔
    *   **Gazebo**: `libgazebo_ros_init.so`، `libgazebo_ros_factory.so`، `libgazebo_ros_p3d.so` وغیرہ کے لیے چیک کریں۔ یقینی بنائیں کہ وہ `<plugin>` ٹیگز کے ذریعے دنیا یا ماڈل فائلز میں، یا `gzserver` کمانڈ لائن آرگومنٹس کے ذریعے لوڈ ہو رہے ہیں۔
    *   **Isaac Sim**: یقینی بنائیں کہ `omni.isaac.ros2_bridge` ایکسٹینشن Isaac Sim میں فعال ہے۔ متعلقہ پریمز کے ساتھ ROS 2 کمپونینٹس (مثلاً `ROS2Camera`، `ROS2DiffDrive`) کے لیے USD اسٹیج چیک کریں۔
    *   **Unity**: یقینی بنائیں کہ ROS-TCP-Connector یا ROS-Unity-Integration پیکیج آپ کے Unity پروجیکٹ میں صحیح طریقے سے کنفیگر ہے، اور `ROSConnection` GameObject موجود ہے اور صحیح IP/Port کے ساتھ کنفیگر ہے۔
2.  **روبوٹ ماڈل کنفیگریشن چیک کریں**: Gazebo/Isaac Sim کے لیے، یقینی بنائیں کہ آپ کے روبوٹ کا URDF/SDF/USD صحیح طریقے سے جوائنٹس اور لنکس کی وضاحت کرتا ہے، اور کوئی بھی ROS 2 کنٹرول پلگ انز صحیح طریقے سے متعین کیے گئے ہیں۔ Isaac Sim کے لیے، یقینی بنائیں کہ روبوٹ ایک `Articulation` ہے اور ROS 2 کمپونینٹس صحیح پریمز سے منسلک ہیں۔
3.  **ٹاپک نیم چیک کریں**: یقینی بنائیں کہ شبیہ کے ذریعہ شائع کردہ ٹاپک نیم (مثلاً `/camera/image_raw`، `/scan`، `/cmd_vel`) ROS 2 نوڈز کے متوقع ٹاپک نیم سے مماثل ہیں۔ تصدیق کے لیے `ros2 topic list` اور `ros2 topic echo <topic_name>` استعمال کریں۔
4.  **شبیہ کا وقت**: اگر کنٹرول بے ترتیب ہے، تو شبیہ کی اپ ڈیٹ کی شرح چیک کریں اور یقینی بنائیں کہ آپ کا ROS 2 کنٹرول لوپ فریکوئینسی مناسب ہے۔

## LLM/قدرتی زبان کی پروسیسنگ کی یکجہتی کے مسائل

### علامات
*   LLM جواب نہیں دیتا یا غیر متعلقہ جوابات دیتا ہے۔
*   قدرتی زبان کے حکم کی پروسیسنگ میں زیادہ تاخیر۔
*   API خرابیاں (مثلاً شرح کی حدود، تصدیق)۔

### حل
1.  **API کیز اور کنفیگریشن**: دو بار چیک کریں کہ آپ کا OpenAI API کی (یا دیگر LLMs کے لیے کیز) صحیح طریقے سے ماحولیاتی متغیر (`OPENAI_API_KEY`) کے طور پر سیٹ کیا گیا ہے یا ایک محفوظ کنفیگریشن فائل سے لوڈ کیا گیا ہے۔
2.  **شرح کی حدود کو سنبھالیں**: API کالز کے لیے ایکسپونینشل بیک آف کے ساتھ دوبارہ کوشش کی منطق نافذ کریں۔ عام کمانڈز کے لیے جوابات کی کیشے کو ذخیرہ کرنا غور کریں۔
    ```python
import openai
import time
import backoff # pip install backoff

@backoff.on_exception(backoff.expo, openai.RateLimitError)
def get_completion_with_backoff(**kwargs):
    return openai.ChatCompletion.create(**kwargs)

# openai.ChatCompletion.create کی بجائے get_completion_with_backoff استعمال کریں
    ```
3.  **پروموٹ انجینئرنگ**: LLM کے لیے اپنے پروموٹس کو مزید مخصوص اور ساخت یافتہ بنانے کے لیے بہتر بنائیں۔ مثال کے طور پر، ایک صاف سیاق (آپ ایک ہیومنوائڈ روبوٹ کو کنٹرول کر رہے ہیں...)، ان پٹ (صارف کمانڈ یہ ہے: ...)، اور مطلوبہ آؤٹ پٹ فارمیٹ (ایکشنز کی JSON فہرست کے ساتھ جواب دیں: ...) فراہم کریں۔
4.  **مقامی ماڈلز**: اگر تاخیر یا API اخراجات ممنوع ہیں، تو مقامی LLMs (مثلاً `llama.cpp`، `transformers` کے ساتھ ایک ڈاؤن لوڈ کردہ ماڈل کے ذریعے) یا چھوٹے، تیز ماڈلز کو ذہنی منصوبہ بندی کے مرحلے کے لیے استعمال کرنا غور کریں۔

## کمپیوٹر وژن (چیز کا پتہ لگانا) کے مسائل

### علامات
*   کیمرہ امیج میں چیز کا پتہ نہیں چلتا۔
*   چیز کی غلط مقامیت (باؤنڈنگ باکس/کوآرڈینیٹس غلط ہیں)۔
*   ادراک پائپ لائن میں زیادہ تاخیر۔

### حل
1.  **کیمرہ ڈیٹا چیک کریں**: تصدیق کریں کہ آپ کا ROS 2 نوڈ کیمرہ ٹاپک (مثلاً `/camera/rgb/image_raw`) کو صحیح طریقے سے سبسکرائب کر رہا ہے اور امیجز حاصل کر رہا ہے۔ ٹاپک کو براہ راست دیکھنے کے لیے `ros2 run image_view image_view` استعمال کریں۔
2.  **ماڈل کی موزونیت**: یقینی بنائیں کہ آپ کا چیز کا پتہ لگانے والا ماڈل آپ کی شبیہ سازی کے ماحول میں موجود اشیاء پر تربیت یافتہ ہے یا ان کے لیے مناسب ہے۔ حقیقی دنیا کی امیجز پر تربیت یافتہ ماڈل فائن ٹیوننگ یا ڈومین اڈاپٹیشن کے تکنیکوں کے بغیر مصنوعی ڈیٹا پر اچھی طرح کام نہیں کر سکتا۔
3.  **امیج پری پروسیسنگ**: چیک کریں کہ امیج کا فارمیٹ، ریزولوشن، اور رنگ کا علاقہ (مثلاً BGR بمقابلہ RGB) آپ کے ڈیٹیکشن ماڈل کی توقعات سے مماثل ہے۔
4.  **کوآرڈینیٹ فریم ٹرانسفارمیشن**: کیمرہ فریم میں چیز کا پتہ چلانے کے بعد، آپ کو اکثر 2D پکسل کوآرڈینیٹس کو 3D ورلڈ کوآرڈینیٹس میں تبدیل کرنا ضروری ہوتا ہے (مثلاً گہرائی کی معلومات یا کیمرہ کے انٹرنسکس/ایکسٹرنسکس کا استعمال کرتے ہوئے) تاکہ مینیپولیشن کا منصوبہ بنا سکیں۔
5.  **کارکردگی**: اگر ایک بھاری ماڈل استعمال کر رہے ہیں، تو ڈیٹیکشن نوڈ کو الگ تھریڈ پر یا کم فریکوئینسی پر چلانے پر غور کریں۔ ممکن ہو تو ماڈل کو بہتر بنائیں (مثلاً کوانٹائزیشن)۔

## نیویگیشن (Nav2) کے مسائل

### علامات
*   روبوٹ راستہ کا منصوبہ بنانے میں ناکام ہوتا ہے۔
*   روبوٹ پھنس جاتا ہے یا جھومنے لگتا ہے۔
*   روبوٹ رکاوٹوں سے بچنے میں ناکام ہوتا ہے۔

### حل
1.  **نقشہ اور لوکلائزیشن**: یقینی بنائیں کہ آپ کے پاس ایک درست نقشہ ہے اور روبوٹ اس کے اندر درست طریقے سے لوکلائزڈ ہے (`/amcl_pose` چیک کریں یا `rviz2` استعمال کریں)۔
2.  **کوسٹ میپ کنفیگریشن**: اپنے `costmap_2d` پیرامیٹر سیٹ چیک کریں۔ یقینی بنائیں کہ `obstacle_range`، `raytrace_range`، `inflation_radius` وغیرہ آپ کے روبوٹ اور ماحول کے لیے ٹیون کیے گئے ہیں۔
3.  **پلانر کنفیگریشن**: ڈیفالٹ `NavFn` گلوبل پلانر یا `DWB` مقامی پلانر بہترین نہیں ہو سکتا۔ دیگر متبادلس جیسے `GlobalPlanner`، `TEBPlanner`، یا `Smoother` دستیاب ہونے پر تجربہ کریں۔
4.  **روبوٹ فوٹ پرنٹ**: یقینی بنائیں کہ روبوٹ کا فوٹ پرنٹ (کوسٹ میپ کنفیگ میں `robot_radius` یا `footprint` پیرامیٹر) تصادم سے بچاؤ کے لیے اس کے جسمانی سائز کو صحیح طریقے سے ظاہر کرتا ہے۔
5.  **TF ٹری**: یقینی بنائیں کہ TF ٹری صحیح اور مکمل ہے (مثلاً `map` -> `odom` -> `base_link` -> `camera`، `laser`)۔ `ros2 run tf2_tools view_frames` اور `ros2 run tf2_ros tf2_echo <frame1> <frame2>` استعمال کریں۔

## مینیپولیشن کے مسائل

### علامات
*   روبوٹ بازو حکم کے مطابق حرکت نہیں کرتا۔
*   گریسنگ مسلسل ناکام ہوتی ہے۔
*   انورس کنیمیٹکس (IK) سالور ناکام ہوتا ہے۔

### حل
1.  **کنٹرول انٹرفیس**: یقینی بنائیں کہ آپ صحیح کنٹرول انٹرفیس استعمال کر رہے ہیں (مثلاً جوائنٹ ٹریجکٹری کنٹرولر، MoveIt! موشن پلاننگ)۔ کنٹرولر مینیجر کی حیثیت `ros2 control list_controllers` کے ساتھ چیک کریں۔
2.  **جوائنٹ حدود**: یقینی بنائیں کہ کمانڈ کردہ جوائنٹ پوزیشنز آپ کے روبوٹ کے URDF/SDF میں وضاحت کردہ جسمانی حدود کے اندر ہیں۔
3.  **گریسنگ کی حکمت عملی**: گریسنگ ایک پیچیدہ کام ہے۔ سادہ، جیومیٹریکلی گریسنگ والی اشیاء کے ساتھ شروع کریں۔ دستیاب ہونے پر ایک پری ڈیفائنڈ گریس گینریٹر یا ایک سیکھا ہوا گریس ماڈل استعمال کرنا غور کریں۔
4.  **شبیہ فزکس**: Gazebo میں، یقینی بنائیں کہ چیز اور گرپر کے لیے `ODE` یا `Bullet` فزکس پیرامیٹر حقیقی ہیں (ماس، فریکشن، سٹفنس) تاکہ مستحکم گریسنگ کی اجازت مل سکے۔ Isaac Sim میں، PhysX سیٹنگس چیک کریں۔
5.  **اینڈ ایفیکٹر کی سیدھ**: چیز کے متعلق اینڈ ایفیکٹر کی درست پوزیشننگ اور سمت گریسنگ کامیابی کے لیے اہم ہے۔ مطلوبہ اینڈ ایفیکٹر پوز کے لیے جوائنٹ اینگلز کا حساب لگانے کے لیے MoveIt! یا کسٹم IK سالور استعمال کریں۔

## کارکردگی کے مسائل

### علامات
*   مجموعی سسٹم سست چلتا ہے۔
*   شبیہ کا وقت فیکٹر کم ہے (< 1.0)۔
*   زیادہ CPU/GPU استعمال۔

### حل
1.  **شبیہ کی ترتیبات**: Gazebo/Isaac Sim میں، فزکس اپ ڈیٹ کی شرح، رینڈرنگ کی معیار (Isaac Sim میں)، یا غیر ضروری پلگ انز/رینڈرنگ کو غیر فعال کریں اگر ریل ٹائم کارکردگی ترقی کے لیے اہم نہیں ہے۔
2.  **LLM/ادراک کی فریکوئینسی**: بھاری AI ماڈلز (LLM کالز، چیز کا پتہ لگانا) کو زیادہ سے زیادہ شبیہ کی فریکوئینسی پر نہ چلائیں۔ کم فریکوئینسی (مثلاً 1-5 Hz) استعمال کریں یا انہیں مخصوص واقعات کی بنیاد پر محرک کریں۔
3.  **ROS 2 نوڈ کی فریکوئینسی**: یقینی بنائیں کہ آپ کے کنٹرول نوڈز مناسب فریکوئینسی پر چل رہے ہیں۔ بے حد زیادہ شائع/سبسکرائب کرنے سے غیر ضروری اوور ہیڈ پیدا ہو سکتا ہے۔
4.  **ہارڈ ویئر**: یقینی بنائیں کہ آپ کی ترقی کی مشین کم از کم تقاضوں کو پورا کرتی ہے، خاص طور پر Isaac Sim (NVIDIA GPU کے ساتھ RTX) اور پیچیدہ ادراک ماڈلز چلانے کے لیے۔

ان علاقوں کو منظم طریقے سے چیک کر کے، آپ کو اپنے کیپسٹون پروجیکٹ میں زیادہ تر یکجہتی کی چیلنجز کو پہچاننے اور حل کرنے میں کامیابی ملنا چاہیے۔
